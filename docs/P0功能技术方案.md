## 1. 方案概述

### 1.1 P0功能范围

根据《功能需求评估报告》的综合评分，P0功能包括：

| 模块 | 综合评分 | 开发优先级 | 说明 |
|------|---------|-----------|------|
| **模块1: 需求分析模块** | 9.28 | 🟢 最高优先级 | 战略核心，必做 |
| **模块6: 知识问答模块** | 8.08 | 🟢 第二优先级 | 激活知识库价值 |

### 1.2 核心目标

**需求分析模块**：
- 帮助用户团队准确识别企业业务部门的Ariba需求
- 形成结构化需求文档和用户故事
- 基于知识库标签实现需求与Ariba能力匹配
- 提供周边能力识别与成本评估

**知识问答模块**：
- 基于知识库（28个文档）回答用户Ariba相关问题
- 提供带来源标注的答案
- 标注适用范围（模块/版本/集成依赖）
- 识别并告知周边能力需求

### 1.3 技术原则

| 原则 | 说明 |
|------|------|
| **复用优先** | 最大化利用现有代码和知识库 |
| **模块化** | 每个模块独立可测试、可部署 |
| **可扩展** | 支持后续模块（蓝图设计、配置指导等）无缝接入 |
| **渐进式** | MVP优先，快速迭代验证 |

## 3. 功能模块设计

### 3.1 需求分析模块（模块1）

#### 3.1.1 核心流程

```
用户输入需求 → 意图识别 → 引导式访谈 → 能力匹配 → 
需求结构化 → 优先级评估 → 文档生成 → 用户故事生成
```

#### 3.1.2 子功能清单

| 功能ID | 功能名称 | 优先级 | 输入 | 输出 | 技术方案 |
|--------|----------|--------|------|------|----------|
| **RA-F01** | 需求收集引导 | P0 | 自然语言描述 | 结构化需求字段 | Coze意图识别 + 动态表单 |
| **RA-F02** | Ariba能力匹配 | P0 | 需求描述 | 匹配的能力列表 + 标签 | 向量检索 + 规则匹配 |
| **RA-F03** | 需求文档生成 | P0 | 需求列表 | Markdown文档 | LLM生成 + 模板化 |
| **RA-F04** | 用户故事生成 | P0 | 需求列表 | User Story列表 | 模板化生成 |
| **RA-F05** | 需求优先级评估 | P1 | 需求列表 | RICE评分 + KANO分类 | 算法计算 + LLM辅助 |
| **RA-F06** | 周边能力识别 | P0 | 需求列表 | 周边能力列表 + 成本 | 知识库标签提取 + 成本库 |

#### 3.1.3 API端点设计

| 端点 | 方法 | 功能 | 参数 | 响应 |
|------|------|------|------|------|
| `/api/requirement/interview/start` | POST | 开始需求访谈 | `{ industry?: string }` | `{ sessionId, firstQuestion }` |
| `/api/requirement/interview/next` | POST | 下一个问题 | `{ sessionId, answer }` | `{ question, options, progress }` 或 `{ status: 'completed', responses }` |
| `/api/requirement/match` | POST | 能力匹配 | `{ description, modules: string[] }` | `{ matched: { tag: capabilities[] } }` |
| `/api/requirement/document` | POST | 生成需求文档 | `{ requirements, context }` | `{ documentId, markdown, html }` |
| `/api/requirement/user-stories` | POST | 生成用户故事 | `{ requirements, context }` | `{ stories, summary }` |
| `/api/requirement/priority` | POST | 优先级评估 | `{ requirements }` | `{ evaluated: requirements[] }` |
| `/api/requirement/peripheral` | POST | 周边能力识别 | `{ requirements, context }` | `{ capabilities, summary, recommendations }` |

### 3.2 知识问答模块（模块6）

#### 3.2.1 核心流程

```
用户提问 → 意图识别 → 知识检索（RAG） → 答案生成 → 
来源标注 → 适用范围标注 → 周边能力识别 → 响应
```

#### 3.2.2 子功能清单

| 功能ID | 功能名称 | 优先级 | 输入 | 输出 | 技术方案 |
|--------|----------|--------|------|------|----------|
| **KQ-F01** | 知识库语义检索 | P0 | 自然语言问题 | 相关文档片段 + 相似度 | RAG + 向量数据库 |
| **KQ-F02** | 答案生成 | P0 | 问题 + 检索结果 | 自然语言答案 + 来源 | LLM生成 |
| **KQ-F03** | 来源标注 | P0 | 答案 + 文档片段 | 带引用的答案 | 元数据提取 |
| **KQ-F04** | 适用范围标注 | P1 | 答案 + 文档元数据 | 适用范围提示 | 规则 + LLM |
| **KQ-F05** | 周边能力识别 | P1 | 答案 + 文档标签 | 周边能力提示 | 标签提取 |
| **KQ-F06** | 对话历史管理 | P2 | 会话上下文 | 上下文感知的回答 | 会话管理 |

#### 3.2.3 API端点设计

| 端点 | 方法 | 功能 | 参数 | 响应 |
|------|------|------|------|------|
| `/api/knowledge/query` | POST | 知识问答 | `{ question, sessionId?, context? }` | `{ answer, sources, applicableRange, peripheralCapabilities, confidence }` |
| `/api/knowledge/search` | POST | 知识检索 | `{ query, topK?, filter? }` | `{ results: [] }` |
| `/api/knowledge/sessions/:id` | GET | 获取会话历史 | - | `{ messages: [] }` |

## 5. 知识库检索方案

### 5.1 RAG架构设计

```
┌─────────────────────────────────────────────────────────┐
│                     文档处理流程                         │
├─────────────────────────────────────────────────────────┤
│  1. 知识库扫描                                           │
│     ↓                                                   │
│  2. 文档解析 (markitdown)                               │
│     ↓                                                   │
│  3. 文本分块 (chunk_size: 500-1000 tokens)              │
│     ↓                                                   │
│  4. 元数据提取 (标签、模块、版本、来源)                  │
│     ↓                                                   │
│  5. 向量嵌入 (OpenAI Embeddings)                        │
│     ↓                                                   │
│  6. 存储到向量数据库 (ChromaDB)                          │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                     检索流程                             │
├─────────────────────────────────────────────────────────┤
│  1. 用户提问                                             │
│     ↓                                                   │
│  2. 意图识别                                             │
│     ↓                                                   │
│  3. 查询向量嵌入                                         │
│     ↓                                                   │
│  4. 向量相似度检索 (Top-K)                               │
│     ↓                                                   │
│  5. 重排序 (Hybrid: vector + keyword)                   │
│     ↓                                                   │
│  6. 上下文构建                                          │
│     ↓                                                   │
│  7. LLM生成答案                                         │
│     ↓                                                   │
│  8. 来源标注 + 适用范围标注                              │
└─────────────────────────────────────────────────────────┘
```

### 5.2 知识库初始化流程

```javascript
// src/services/knowledge/init.js

const VectorDB = require('./vector-db');

async function initializeKnowledgeBase() {
```

### 6.2 项目上下文服务

```javascript
// src/services/project/context-service.js

class ProjectContextService {
  constructor() {
    this.projectsPath = path.join(__dirname, '../../data/projects');
    this.ensureDirectory();
  }
  
  async createProject(projectData) {
    const project = {
      projectId: this.generateId(),
      name: projectData.name,
      industry: projectData.industry,
      modules: projectData.modules || [],
      existingSystem: projectData.existingSystem || '无',
      users: projectData.users || 0,
      timeline: projectData.timeline || '未定',
      createdAt: new Date().toISOString(),
      updatedAt: new Date().toISOString(),
      status: 'active'
    };
    
    await this.saveProject(project.projectId, project);
    return project;
  }
  
  async updateContext(projectId, context) {
    const project = await this.getProject(projectId);
    if (!project) {
      throw new Error(`项目 ${projectId} 不存在`);
    }
    
    const projectContext = await this.getProjectContext(projectId);
    const updatedContext = {
      ...projectContext,
      ...context,
      updatedAt: new Date().toISOString()
    };
    
    await this.saveProjectContext(projectId, updatedContext);
    return updatedContext;
  }
  
  generateId() {
    return `PROJ-${Date.now()}`;
  }
}

module.exports = new ProjectContextService();
```

## 7. 技术选型建议

### 7.1 向量数据库对比

| 特性 | ChromaDB | Pinecone | Qdrant |
|------|----------|---------|--------|
| **部署** | 本地/云端 | 云端 | 本地/云端 |
| **成本** | 免费（本地） | 付费（$70/月起） | 免费/付费 |
| **性能** | 中等 | 高 | 高 |
| **易用性** | 高 | 高 | 中等 |
| **推荐场景** | 小规模知识库 | 大规模生产 | 需要控制成本 |

**推荐选择**: ChromaDB（本地部署）
- 免费且无限制
- 28个文档规模足够
- 便于维护和调试

### 7.2 LLM模型选择

| 模型 | 用途 | 成本 | 推荐度 |
|------|------|------|--------|
| **GPT-4o** | 复杂任务（文档生成） | $5/1M tokens | ⭐⭐⭐⭐⭐ |
| **GPT-4o-mini** | 一般任务（问答、分类） | $0.15/1M tokens | ⭐⭐⭐⭐ |
| **text-embedding-3-small** | 向量嵌入 | $0.02/1M tokens | ⭐⭐⭐⭐⭐ |

**推荐组合**:
- 文档生成: GPT-4o（质量要求高）

| 组件 | 选项A | 选项B | 推荐 |

```javascript
// src/services/coze/webhook-handler.js

async function handleCozeWebhook(webhookData) {
  const { event, message } = webhookData;
  
  switch (event) {
    case 'requirement_interview':
      return handleRequirementInterview(message);
    case 'knowledge_query':
      return handleKnowledgeQuery(message);
    default:
      return handleGeneralMessage(message);
  }
}

async function handleKnowledgeQuery(message) {
  const { question, sessionId } = message;
  return await ragService.query(question, { sessionId });
}
```
