# Ariba实施助手 - 开发需求评估方法

## 概述

本文档定义了Ariba实施助手项目的需求评估流程，确保每个开发需求都经过严格评估，避免资源浪费和方向偏离。

---

## 评估流程

```
需求提出
    ↓
┌─────────────────────────────────────────┐
│ 第1步: GSTACK CEO评估                     │
│ 评估：是否符合项目目标？                   │
│ 结果：通过/不通过                         │
└─────────────────────────────────────────┘
    ↓ 通过
┌─────────────────────────────────────────┐
│ 第2步: RICE+KANO评分                     │
│ 量化评估需求价值 + 需求类型分类             │
└─────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────┐
│ 第3步: Ralph多角色评估                   │
│ CEO + PM + Tech Lead + Designer + Dev   │
└─────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────┐
│ 第4步: 综合决策                          │
│ 加权计算 → 优先级排序 → 决策              │
└─────────────────────────────────────────┘
```

## 第1步: GSTACK CEO评估

### 项目目标（北极星）

**Ariba实施助手的目标**：
> 帮助企业IT团队自助完成Ariba系统实施，降低实施门槛，提升实施效率。

### 评估标准

| 问题 | 通过条件 | 不通过条件 |
|------|---------|-----------|
| 是否服务IT实施团队？ | ✅ 是IT团队使用 | ❌ 是业务部门使用 |
| 是否帮助实施Ariba？ | ✅ 实施阶段使用 | ❌ 运营阶段使用 |
| 是否降低实施门槛？ | ✅ 让非专家也能做 | ❌ 需要专家才能用 |
| 是否提升实施效率？ | ✅ 减少时间/成本 | ❌ 增加工作量 |

### 评估模板

```markdown
## 需求：[需求名称]

### GSTACK CEO评估

| 评估项 | 结果 | 说明 |
|--------|------|------|
| 目标用户 | ✅/❌ | IT实施团队 |
| 使用阶段 | ✅/❌ | 实施阶段 |
| 门槛降低 | ✅/❌ | [说明] |
| 效率提升 | ✅/❌ | [说明] |

### CEO决策
- [ ] 通过 → 进入第2步
- [ ] 不通过 → 说明原因，需求退回
```

## 第2步: RICE+KANO评估

### RICE评分框架

| 维度 | 说明 | 评分标准 |
|------|------|---------|
| **R - Reach（覆盖面）** | 多少用户会用到？ | 1-10分 |
| **I - Impact（影响力）** | 对用户价值多大？ | 1-10分 |
| **C - Confidence（信心度）** | 我们有多大把握？ | 50%/80%/100% |
| **E - Effort（工作量）** | 需要多少投入？ | 人天/人周 |

**RICE分数计算**：
```
RICE = (R × I × C) / E
```

### KANO需求分类

| 类型 | 符号 | 特征 | 优先级 |
|------|------|------|--------|
| **基本型需求（Must-be）** | M | 必须有，没有用户会非常不满 | ⭐⭐⭐⭐⭐ 最高 |
| **期望型需求（Performance）** | P | 越多越好，做得越好用户越满意 | ⭐⭐⭐⭐ 高 |
| **魅力型需求（Attractive）** | A | 没有不会不满，有了会很惊喜 | ⭐⭐⭐ 中 |
| **无差异需求（Indifferent）** | I | 有没有都无所谓 | ⭐⭐ 低 |
| **反向需求（Reverse）** | R | 有了反而不满意 | ⭐ 不做 |

### KANO分类方法

通过用户调研确定需求类型：

**问卷设计**：
1. 如果有这个功能，你会满意吗？（正向问题）
2. 如果没有这个功能，你会不满意吗？（反向问题）

**分类判定**：

| 正向回答 | 反向回答 | KANO类型 |
|---------|---------|---------|
| 喜欢 | 喜欢 | A（魅力型）|
| 喜欢 | 理所当然 | A（魅力型）|
| 喜欢 | 无所谓 | I（无差异）|
| 喜欢 | 不喜欢 | R（反向）|
| 理所当然 | 喜欢 | P（期望型）|
| 理所当然 | 理所当然 | M（基本型）|
| 理所当然 | 无所谓 | I（无差异）|
| 无所谓 | 喜欢 | P（期望型）|
| 无所谓 | 理所当然 | M（基本型）|
| 无所谓 | 无所谓 | I（无差异）|
| 不喜欢 | 喜欢 | Q（可疑结果）|
| 不喜欢 | 理所当然 | Q（可疑结果）|

### KANO优先级系数

| KANO类型 | 系数 | 说明 |
|---------|------|------|
| M（基本型） | 2.0 | 必须做，否则产品不可用 |
| P（期望型） | 1.5 | 做得越好，用户越满意 |
| A（魅力型） | 1.2 | 锦上添花，差异化竞争 |
| I（无差异） | 0.5 | 可做可不做 |
| R（反向） | 0 | 不做 |

### 综合评分

```
第2步总分 = RICE分数 × KANO系数
```

### 评估模板

```markdown
## RICE评分

| 维度 | 分数 | 说明 |
|------|------|------|
| R - Reach | /10 | [用户覆盖比例] |
| I - Impact | /10 | [价值说明] |
| C - Confidence | /100% | [信心说明] |
| E - Effort | 人天 | [工作量估算] |

**RICE分数**: (R × I × C) / E = [计算结果]

## KANO分类

**正向问题**：如果有这个功能，你会满意吗？
- [ ] 喜欢 / 理所当然 / 无所谓 / 不喜欢

**反向问题**：如果没有这个功能，你会不满意吗？
- [ ] 喜欢 / 理所当然 / 无所谓 / 不喜欢

**KANO类型**: [M/P/A/I/R]
**KANO系数**: [2.0/1.5/1.2/0.5/0]

**第2步总分**: RICE × KANO系数 = [分数]
```

## 第3步: Ralph多角色评估

### 评估角色

| 角色 | 关注点 | 评估维度 |
|------|--------|---------|
| **CEO** | 商业价值、战略一致性 | 目标对齐、ROI |
| **产品经理（PM）** | 用户价值、产品路线 | 用户体验、功能完整性 |
| **技术负责人（Tech Lead）** | 技术可行性、架构影响 | 技术债务、扩展性 |
| **设计师（Designer）** | 用户体验、交互设计 | 易用性、一致性 |
| **开发者（Developer）** | 实现复杂度、维护成本 | 代码质量、测试成本 |

## 第4步: 综合决策

### 分数汇总

| 评估步骤 | 权重 | 分数 | 加权分数 |
|---------|------|------|---------|
| 第2步: RICE+KANO | 40% | /10 | - |
| 第3步: Ralph多角色 | 60% | /10 | - |
| **总分** | - | - | [最终分数] |

### 决策标准

| 分数区间 | 决策 | 行动 |
|---------|------|------|
| **8-10分** | 🟢 立即开发 | 进入开发队列，优先安排资源 |
| **6-7.9分** | 🟡 排队等待 | 加入需求池，按优先级排序 |
| **4-5.9分** | 🟠 暂缓评估 | 需要更多信息或需求优化 |
| **0-3.9分** | 🔴 拒绝/搁置 | 不符合项目方向，记录原因后关闭 |

## KANO在产品规划中的应用

### 产品生命周期配合

| 阶段 | 重点KANO类型 | 策略 |
|------|-------------|------|
| **MVP阶段** | M（基本型） | 先做必须有，保证产品可用 |
| **成长期** | P（期望型） | 持续优化核心功能，提升满意度 |
| **成熟期** | A（魅力型） | 增加差异化功能，建立竞争壁垒 |

### 资源分配建议

| KANO类型 | 资源占比 | 说明 |
|---------|---------|------|
| M（基本型） | 40% | 必须保证质量 |
| P（期望型） | 35% | 核心竞争力 |
| A（魅力型） | 20% | 差异化创新 |
| I（无差异） | 5% | 视资源情况 |

## 使用说明

### 何时使用

- 任何新功能开发需求
- 用户提出的Feature Request
- 技术债务处理需求
- 第三方集成需求

### 谁来评估

- 主Agent（阿虾）执行评估流程
- 必要时召集Sub-Agent扮演不同角色
- 最终决策需要主人确认

### 评估频率

- 收到新需求时立即评估
- 每周review一次需求池优先级
- 每月复盘评估方法有效性
